# GitHub PR Comment Submission

Automated submission of Fixium code review findings as inline comments on GitHub Pull Requests.

## Overview

This system converts code review findings from JSON format into formatted GitHub PR comments, posting them as inline reviews on specific lines of code.

## Features

- ✅ **Line-specific comments** - Comments appear on exact lines with issues
- 🎨 **Formatted markdown** - Severity badges, type icons, and code blocks
- 📦 **Batch submission** - Efficient API usage with batched requests
- 🔍 **Smart filtering** - Filter by severity, type, or exclude criteria
- 🧪 **Dry-run mode** - Preview comments before submitting
- 🔄 **Retry logic** - Automatic retry with exponential backoff
- ⚡ **Rate limit handling** - Respects GitHub API limits
- 📊 **Summary reporting** - Detailed submission statistics

## Quick Start

### 1. Setup GitHub Token

Create a GitHub Personal Access Token with these scopes:
- `repo` (full control)
- `write:discussion` (for PR comments)

Create token at: https://github.com/settings/tokens

### 2. Configure Environment

Edit `config/github.env`:

```bash
GITHUB_TOKEN="ghp_xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
GITHUB_OWNER="your-organization"
GITHUB_REPO="pay-go-metrics-monitor"
```

Or set environment variables:

```bash
export GITHUB_TOKEN="ghp_xxxxxxxxxxxx"
export GITHUB_OWNER="your-org"
export GITHUB_REPO="pay-go-metrics-monitor"
```

### 3. Submit Comments

```bash
# Submit all findings to PR #123
./review_workflow.sh submit review1.json 123

# Or use the script directly
./submit_pr_comments.sh review1.json 123
```

## Usage

### Review a Pull Request

The easiest way to review a PR is using the `review-pr` command:

```bash
# Review PR #123 and generate review JSON
./review_workflow.sh review-pr 123

# Review and auto-submit comments
./review_workflow.sh review-pr 123 review_pr123.json --auto-submit

# Review with custom output file
./review_workflow.sh review-pr 123 my_review.json
```

This command will:
1. Fetch the list of changed files from the PR
2. Run Fixium review on those files
3. Generate a review JSON file
4. Optionally submit comments back to the PR (with `--auto-submit`)

### Submit Comments to PR

```bash
# Submit all findings
./submit_pr_comments.sh review1.json 123

# Preview without submitting (dry-run)
./submit_pr_comments.sh review1.json 123 --dry-run

# Only high severity findings
./submit_pr_comments.sh review1.json 123 --severity high

# Only bugs and security issues
./submit_pr_comments.sh review1.json 123 --type bug,security

# Exclude low severity findings
./submit_pr_comments.sh review1.json 123 --exclude-severity low

# Custom batch size
./submit_pr_comments.sh review1.json 123 --batch-size 20
```

### Integration with Review Workflow

```bash
# Full workflow: review, implement, verify, and submit to PR
./review_workflow.sh review @files.txt review1.json
./review_workflow.sh implement review1.json implementation1.json
./review_workflow.sh verify implementation1.json review1.json review2.json
./review_workflow.sh submit review2.json 123
```

## Comment Format

Each finding is formatted as a GitHub markdown comment:

```markdown
🔴 **HIGH SEVERITY** | 🐛 **BUG**

**Issue:** Missing parameter in WHERE clause causes compilation error

**Details:** The WHERE clause has 'CreatedAt <= ?' but jobStartTime is not 
passed as a parameter to the Where() method. This will cause a runtime error 
or incorrect query execution.

**Suggestion:**
```go
Where("SCSendStatus IN(?, ?) AND CreatedAt <= ?", 
      models.UsageSCSendStatus.NOTSENT, 
      models.UsageSCSendStatus.FILE_SEND_FAILED, 
      jobStartTime)
```

---
*🤖 Generated by Fixium Code Review | Line 47*
```

### Severity Badges

- 🔴 **HIGH** - Critical issues requiring immediate attention
- 🟡 **MEDIUM** - Important issues to address
- 🔵 **LOW** - Minor improvements or suggestions

### Type Icons

- 🐛 **BUG** - Code defects or errors
- 🔒 **SECURITY** - Security vulnerabilities
- 🔧 **MAINTAINABILITY** - Code quality improvements
- ⚡ **PERFORMANCE** - Performance optimizations

## Library Components

### 1. Comment Formatter (`lib/comment_formatter.sh`)

Converts JSON findings to formatted markdown comments.

```bash
# Test format a finding
./lib/comment_formatter.sh --test review1.json 0

# Show summary statistics
./lib/comment_formatter.sh --stats review1.json

# List files with findings
./lib/comment_formatter.sh --files review1.json
```

**Functions:**
- `format_finding()` - Format single finding as markdown
- `format_finding_with_code()` - Format with syntax-highlighted code blocks
- `get_severity_badge()` - Get emoji for severity level
- `get_type_icon()` - Get emoji for finding type
- `group_findings_by_file()` - Group findings by file path

### 2. GitHub API Client (`lib/github_api.sh`)

Handles GitHub API interactions.

```bash
# Validate GitHub token
./lib/github_api.sh --validate-token

# Check API rate limit
./lib/github_api.sh --rate-limit

# Get PR information
./lib/github_api.sh --pr-info owner repo 123

# Get PR commit SHA
./lib/github_api.sh --pr-sha owner repo 123

# List PR files
./lib/github_api.sh --pr-files owner repo 123
```

**Functions:**
- `validate_token()` - Verify GitHub token is valid
- `check_rate_limit()` - Check API rate limit status
- `get_pr_info()` - Get PR details
- `get_pr_commit_sha()` - Get latest commit SHA
- `get_pr_files()` - List changed files in PR
- `create_review_comment()` - Submit single comment
- `create_review_batch()` - Submit multiple comments efficiently
- `retry_api_call()` - Retry with exponential backoff

## Configuration

### Environment Variables

| Variable | Description | Default |
|----------|-------------|---------|
| `GITHUB_TOKEN` | GitHub personal access token | (required) |
| `GITHUB_OWNER` | Repository owner/organization | (required) |
| `GITHUB_REPO` | Repository name | pay-go-metrics-monitor |
| `GITHUB_API_URL` | GitHub API endpoint | https://api.github.com |
| `MAX_COMMENTS_PER_BATCH` | Comments per batch request | 30 |
| `RATE_LIMIT_BUFFER` | Rate limit safety buffer | 10 |
| `MAX_RETRIES` | Maximum retry attempts | 3 |
| `RETRY_DELAY` | Initial retry delay (seconds) | 5 |

### Configuration File

Create `config/github.env` with your settings:

```bash
GITHUB_TOKEN="ghp_xxxxxxxxxxxx"
GITHUB_OWNER="your-organization"
GITHUB_REPO="pay-go-metrics-monitor"
MAX_COMMENTS_PER_BATCH=30
RATE_LIMIT_BUFFER=10
```

## Filtering Options

### By Severity

```bash

--severity high

# High and medium
--severity high,medium

# Exclude low severity
--exclude-severity low
```

### By Type

```bash
# Only bugs
--type bug

# Bugs and security issues
--type bug,security

# All except maintainability
--exclude-type maintainability
```

### Combined Filters

```bash
# High severity bugs only
./submit_pr_comments.sh review1.json 123 --severity high --type bug

# Medium/high severity, exclude maintainability
./submit_pr_comments.sh review1.json 123 \
  --severity medium,high \
  --exclude-type maintainability
```

## Error Handling

### Automatic Retries

Failed API calls are automatically retried with exponential backoff:
- Attempt 1: Immediate
- Attempt 2: 5 seconds delay
- Attempt 3: 10 seconds delay
- Attempt 4: 20 seconds delay (if MAX_RETRIES=4)

### Rate Limiting

The system monitors GitHub API rate limits:
- Checks rate limit before submission
- Warns when remaining requests < buffer
- Adds delays between batch submissions
- Respects rate limit reset times

### File Validation

Comments are only submitted for files in the PR:
- Validates file exists in PR diff
- Skips files not in PR with warning
- Reports skipped comments in summary

## Troubleshooting

### Token Validation Failed

```bash
# Check token is set
echo $GITHUB_TOKEN

# Validate token
./lib/github_api.sh --validate-token

# Check token scopes at:
# https://github.com/settings/tokens
```

### PR Not Found

```bash
# Verify PR exists and is open
./lib/github_api.sh --pr-info owner repo 123

# Check repository settings
echo $GITHUB_OWNER
echo $GITHUB_REPO
```

### Rate Limit Exceeded

```bash
# Check current rate limit
./lib/github_api.sh --rate-limit

# Wait for rate limit reset
# Or use a different token
```

### File Not in PR

Files not in the PR diff will be skipped:
```
⚠️  File not in PR (skipping): some/file.go
```

This is expected if the review includes files not changed in the PR.

## Best Practices

1. **Use dry-run first** - Preview comments before submitting
2. **Filter appropriately** - Don't overwhelm with low-severity comments
3. **Batch wisely** - Default batch size (30) is optimal for most cases
4. **Monitor rate limits** - Check limits before large submissions
5. **Review PR files** - Ensure review covers files in the PR
6. **Use meaningful filters** - Focus on critical issues first

## Integration Examples

### CI/CD Pipeline

```yaml
# .github/workflows/code-review.yml
name: Code Review
on: pull_request

jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      
      - name: Run Fixium Review
        run: |
          ./review_workflow.sh review @changed_files.txt review.json
      
      - name: Submit PR Comments
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          GITHUB_OWNER: ${{ github.repository_owner }}
          GITHUB_REPO: ${{ github.event.repository.name }}
        run: |
          ./submit_pr_comments.sh review.json ${{ github.event.pull_request.number }} \
            --severity high,medium \
            --exclude-type maintainability
```

### Pre-commit Hook

```bash
#!/bin/bash
# .git/hooks/pre-push

# Review changed files
git diff --name-only origin/main...HEAD > changed_files.txt
./review_workflow.sh review @changed_files.txt review.json

# Submit to PR if exists
if [ -n "$PR_NUMBER" ]; then
    ./submit_pr_comments.sh review.json $PR_NUMBER --severity high
fi
```
